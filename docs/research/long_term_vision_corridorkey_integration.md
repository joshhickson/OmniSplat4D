## **1. Introduction: The Compositing Problem OmniSplat4D Was Not Designed to Solve — But Can**

Modern visual effects compositing pipelines are fundamentally constrained by a two-dimensional ceiling. Regardless of the photorealistic quality of individual elements — actor performance plates, background environments, particle simulations — the act of compositing them together is a pixel-layer stacking operation. A foreground plate is alpha-blended over a background plate. Occlusion between layers is manually managed by rotoscope artists. Parallax between a moving camera and background elements is approximated by 2.5D projections or card-based set extensions. The physically-correct behavior of light — incident environmental illumination reflecting off a foreground subject, or a foreground subject geometrically occluding background elements at varying depths — is either hand-painted or produced by an expensive offline render pipeline requiring dedicated render farms, Houdini simulations, and licensed compositing software.

CorridorKey, authored by Niko Pueringer and publicly available at https://github.com/nikopueringer/CorridorKey.git, represents a state-of-the-art approach to one specific node in this 2D pipeline: green screen alpha unmixing. Rather than producing binary segmentation masks, CorridorKey performs physically-motivated color unmixing — reconstructing the true foreground color and a continuous linear alpha channel per pixel, including semi-transparent edges arising from motion blur, out-of-focus regions, and fine hair detail. The model accepts raw green screen footage and a coarse alpha hint mask (generated by GVM, VideoMaMa, BiRefNet, or a manual matte) and outputs linear EXR passes compatible with professional node-based compositing applications including Nuke, Fusion, and DaVinci Resolve.

CorridorKey solves the 2D layer problem exceptionally well. It does not attempt to solve the three-dimensional compositing problem, because that would require scene geometry — which CorridorKey does not produce or consume.

OmniSplat4D produces scene geometry. Not as a side effect, but as its primary output. The static background is a 3D Gaussian Splat. The dynamic subject is a 4D Gaussian Splat with a continuous temporal deformation field. The compositing layer merges both primitive sets in GPU memory before rasterization, sorted by physical depth. The coordinate systems of both representations are anchored to the same COLMAP world frame.

The long-term vision articulated in this document is directionally simple: **OmniSplat4D provides CorridorKey with what it currently lacks — geometric grounding — and CorridorKey provides OmniSplat4D with what its binary masking pipeline currently lacks — sub-pixel accurate linear alpha.** The integration path is bidirectional. The result, at maturity, is a compositing workflow that is depth-correct, parallax-correct, physically-lit, and operable on a single consumer GPU without cloud dependency at playback time.

This vision, if realized, would represent a meaningful shift in what is accessible to independent VFX practitioners: a production-quality volumetric compositing pipeline operable outside of a studio infrastructure.

## **2. CorridorKey: Capabilities, Constraints, and Evolving State**

### **2.1. What CorridorKey Currently Does**

At the time of this document's authorship, CorridorKey implements a neural green screen unmixing model trained to separate foreground and background color contributions at the sub-pixel level. The core inference architecture performs what is formally termed alpha unmixing — solving the compositing equation `C = alpha * F + (1 - alpha) * B` for the unknown foreground color `F` and alpha `alpha`, given the observed composite pixel color `C` and a known (or strongly approximated) background color `B`. Green screen keying is, mathematically, a constrained version of this problem where `B` is a known chromatic constant.

The technical distinction between CorridorKey's approach and naive chroma keying is significant. Naive keying computes a binary or soft mask from the green channel's dominance, then applies it uniformly. This approach fails at semi-transparent boundaries — motion-blurred limbs, defocused hair, translucent fabric — because the compositing equation in those regions mixes foreground and background colors non-trivially. CorridorKey's neural unmixing operates on the full color information to reconstruct the physically-correct pre-multiplied foreground, preserving detail that naive keying destroys.

The model outputs 16-bit or 32-bit linear EXR files for VFX workflow compatibility. It supports multi-backend inference (CUDA, Apple MPS, ROCm, CPU) and handles plates up to 4K resolution via dynamic inference tiling over a 2048×2048 backbone.

### **2.2. What CorridorKey Does Not Do**

CorridorKey is a per-frame 2D color separation tool. Its four structural constraints — each addressable by OmniSplat4D — are:

**Temporal independence.** Each frame is processed independently. There is no temporal model enforcing consistency of the alpha channel between frames. Alpha values at silhouette edges can flicker between frames even when the underlying subject motion is smooth, because the neural network's output is not constrained by any inter-frame geometric relationship. This temporal incoherence is a known artifact class in per-frame keying systems.

**Spill correction is a 2D heuristic.** Green screen spill — the green light reflected from the background onto the subject's hair, skin, and clothing — is currently corrected via hue rotation and color grading in compositing. This is a per-pixel 2D approximation. The physically-correct approach requires knowing the 3D geometry of the subject, the position and color of the background light source, and the subject's surface properties. CorridorKey operates without any of this information.

**Compositing is layer stacking.** CorridorKey's output is a 2D RGBA plate. When composited over a background, the layer relationship is fixed: foreground is always in front of background. A subject walking behind a pillar, a hand reaching into a practical set element, or any scene requiring depth-correct occlusion between foreground and background elements cannot be handled by a 2D compositing operation. It requires rotoscope, a 3D render, or a depth-buffer-based compositing node.

**Green screen is required.** The unmixing model's ability to separate foreground from background depends on the chromatic constancy of the background. Without a controlled, uniformly lit green screen environment, the color unmixing problem becomes severely underconstrained. CorridorKey cannot be applied to arbitrary real-world footage.

### **2.3. The Evolving Nature of CorridorKey**

It is critical for any implementation effort derived from this document to acknowledge that CorridorKey is an actively maintained and evolving project. The repository at https://github.com/nikopueringer/CorridorKey.git continues to receive updates, architectural revisions, and community forks as the broader field of neural matting advances. The specific inference architecture, the format of hint mask inputs, the output EXR schema, and the supported hardware backends may change substantially between this document's authorship and any future integration attempt. Before any code is written against CorridorKey's API or output format, the integration author must:

1. Pull the current HEAD of the CorridorKey repository and read its documentation.
2. Survey the active forks to determine whether a community fork has diverged meaningfully in capability or approach.
3. Evaluate whether intervening advances in general video matting (such as improvements to GVM, VideoMaMa, BiRefNet, or successor models) have changed the optimal integration point.

This document deliberately avoids specifying API call signatures, input format assumptions, or model checkpoint paths for CorridorKey. Those details are transient. The architectural integration logic described in Sections 3–6 is written in terms of data semantics — what information flows between systems — not in terms of current implementation specifics.

## **3. Symmetry Analysis: What Each System Provides the Other**

The most efficient framing of this integration is not additive but complementary. Each system provides the input the other system currently must approximate.

| Capability | CorridorKey Today | OmniSplat4D Today | Integrated System |
|---|---|---|---|
| Alpha matte quality | Continuous linear alpha via neural unmixing | Binary mask via SAM 2.1 | CorridorKey replaces SAM at Phase 1/2 masking |
| Background model | Assumed chromatic constant (green screen) | Explicit 3DGS geometry | OmniSplat4D 3DGS replaces green screen requirement |
| Temporal coherence | Per-frame independent inference | SWinGS temporal consistency loss | 4DGS opacity field provides geometrically-grounded alpha |
| Compositing dimension | 2D layer stacking | Pre-rasterization Gaussian merge + depth sort | Depth-correct occlusion, parallax, free viewpoint |
| Spill correction | 2D heuristic color grading | N/A | 3DGS background lighting enables physical relighting |
| Hardware requirement at playback | Offline render pipeline | ONNX + SPZ — no cloud at playback | Unchanged — consumer GPU runtime |

The integration is not a merger of two products. It is a pipeline extension where OmniSplat4D's Phase 1 output (the static 3DGS background) becomes the input that CorridorKey's hint generation stage currently must approximate with a chromatic constant, and CorridorKey's alpha output becomes the mask input that OmniSplat4D's Phase 1 and Phase 2 masking stages currently produce with a binary segmentation model.

## **4. Integration Component 1 — Green-Screen-Free Hint Generation**

### **4.1. Concept**

CorridorKey's inference model does not require the hint mask to be derived from green screen. The hint is a coarse alpha estimate — a rough indication of which pixels belong to the foreground subject. The model's neural unmixing process refines this coarse estimate into a sub-pixel accurate linear alpha. Green screen is the most common source of this coarse estimate in professional production, but the model is agnostic to its origin.

OmniSplat4D Phase 1 produces a trained 3DGS representation of the static background. Once this reconstruction exists, it is possible to render a synthetic version of the background at any known camera pose and compare it against the original captured frame. Pixels that deviate substantially from the rendered background are dynamic — they belong to a subject that was not present in the static reconstruction. This difference signal is, semantically, a coarse alpha hint. It does not require a green screen because it uses geometric knowledge of the scene rather than chromatic knowledge of the background.

### **4.2. Proposed Pipeline**

1. OmniSplat4D Phase 1 trains the static 3DGS background from clean background frames (frames where the dynamic subject is absent, or masked out by the current SAM-based pipeline).
2. For each frame containing the dynamic subject, `render_frame()` is called with only the static `GaussianCheckpoint` and the known camera pose for that frame. This produces a synthetic background render.
3. A per-pixel photometric difference is computed between the rendered background and the raw captured frame. Regions where the subject occludes the background produce high difference values; static background regions produce near-zero difference.
4. This difference map is thresholded and morphologically cleaned to produce a coarse foreground mask.
5. The coarse mask is passed to CorridorKey as the hint input. CorridorKey refines it into a continuous linear alpha.
6. The linear alpha output from CorridorKey replaces the binary mask currently produced by `segment/inverter.py` for both Phase 1 background training and Phase 2 subject isolation.

### **4.3. Open Research Questions**

- **Photometric gap.** The rendered 3DGS background will not be a perfect reconstruction of the original captured frame — there will be residual reconstruction error from the training process. High-frequency texture, specular reflections, and shadow boundaries in the static scene may produce spurious difference signals that the coarse hint incorrectly attributes to the dynamic subject. The magnitude of this photometric gap on realistic outdoor footage and its effect on CorridorKey's unmixing quality is unknown and must be characterized empirically.

- **Bootstrap problem.** Phase 1 background training currently depends on the SAM-based masking pipeline to exclude the dynamic subject from the initial reconstruction. If the goal is to replace the SAM pipeline with CorridorKey's higher-quality alpha, a circular dependency arises: you need the background model to generate the hint, but you need the hint to generate the background model. This bootstrap problem requires a solution — either a two-pass iterative refinement (coarse SAM mask → initial 3DGS → CorridorKey hint → refined 3DGS), or a design that accepts the SAM masks only for the Phase 1 reconstruction and then uses the 3DGS background exclusively for Phase 2 hint generation.

- **Multi-camera consistency.** OmniSplat4D's eight virtual cameras share a single optical center. The difference-map hint generation would need to be consistent across all eight camera views for a given frame index. Whether CorridorKey's per-frame inference, applied independently to each virtual camera's cropped view, produces consistent alpha across camera boundaries is unknown.

## **5. Integration Component 2 — Temporally Stable Alpha via 4DGS Opacity Fields**

### **5.1. Concept**

CorridorKey produces a per-frame alpha matte. Temporal flicker in this alpha — small inconsistencies in the opacity of semi-transparent regions between consecutive frames — is an inherent limitation of per-frame neural inference without a temporal model.

OmniSplat4D Phase 2 represents the dynamic subject as a 4D Gaussian field with a continuous deformation model. Each Gaussian primitive carries an opacity value. When the gsplat rasterizer accumulates opacity along camera rays (front-to-back transmittance blending), it produces a per-pixel accumulated alpha as a byproduct of standard RGB rendering. This accumulated alpha is not inferred from color — it is derived from the 3D geometry of the Gaussian primitives and is therefore continuous and consistent across frames by construction. The temporal consistency loss enforced across SWinGS window boundaries further constrains the Gaussian opacity values to evolve smoothly in time.

Re-deriving the alpha matte from the 4DGS opacity field — rather than using CorridorKey's per-frame neural estimate — would produce an alpha channel that is geometrically grounded and temporally smooth. This alpha could then be composited back with CorridorKey's refined foreground color estimate, combining the color precision of neural unmixing with the temporal stability of geometric rendering.

### **5.2. Proposed Pipeline**

1. OmniSplat4D Phase 2 trains the 4DGS dynamic subject representation using CorridorKey-derived alpha masks as the training input (per Section 4).
2. For each frame in the sequence, `render_frame()` is called on the dynamic `GaussianCheckpoint` with the deformation field applied, requesting alpha output in addition to RGB.
3. The per-pixel accumulated alpha from the Gaussian rasterizer is extracted as the temporal alpha matte.
4. This geometrically-derived alpha replaces or blends with CorridorKey's per-frame alpha for the final compositing stage.

### **5.3. Open Research Questions**

- **Alpha boundary fidelity vs. geometric alpha fidelity.** CorridorKey's continuous alpha is computed from sub-pixel color unmixing and captures genuine physical transparency — motion blur, defocus, translucent hair. The 4DGS opacity field is a learned geometric approximation that may not represent these effects accurately unless the Gaussian primitives are numerous and well-initialized at fine-detail boundaries. Whether the geometric alpha matches or exceeds the neural alpha at silhouette boundaries, and under what conditions each is preferable, requires empirical evaluation.

- **Fitting semi-transparent regions.** The entropy regularization loss in `train/pruner.py` is designed to drive Gaussian opacities toward binary states (either fully opaque or fully transparent). This is beneficial for hard-surface geometry and reduces artifacts in occluded regions. It may, however, prevent the model from accurately representing genuinely semi-transparent subject regions such as hair and motion blur. Whether the entropy loss weight needs to be tuned or disabled for subject regions near silhouette boundaries is an open question.

- **Alpha blend strategy.** The optimal weighting between CorridorKey's neural alpha and the 4DGS geometric alpha in the final compositing pass is unknown. A simple alpha blend, a depth-weighted blend, or a learned fusion are all possibilities. This is a research question, not an engineering one.

## **6. Integration Component 3 — Depth-Correct Gaussian-Level Compositing**

### **6.1. Concept**

The most structurally significant integration is compositing at the Gaussian level rather than the pixel level. OmniSplat4D Phase 3 already implements this for its internal static/dynamic pipeline: `concat_gaussians()` merges both primitive sets before rasterization, and the unified depth sort (`radix_sort_by_depth()`) produces physically correct occlusion between static background and dynamic subject Gaussians. This is not a 2D layer operation — it is a pre-rasterization 3D sort over the merged primitive set, equivalent in correctness to a ray-traced render without the cost.

The extension to CorridorKey's use case is architectural: the destination compositing background — which CorridorKey's user normally provides as a 2D plate — could itself be represented as a 3DGS reconstruction. If the background environment is a 3DGS and the foreground subject is a 4DGS, and both are merged at the Gaussian level before rasterization, then:

- Occlusion between foreground and background is geometrically correct at arbitrary depth relationships.
- Parallax between foreground and background is derived from the 3D structure of both representations, not from 2.5D card approximations.
- The camera can be moved freely in 3D space and the composite remains valid.
- No rotoscope or per-frame manual occlusion work is required for foreground elements moving behind background geometry.

### **6.2. Proposed Pipeline**

1. The destination background environment is reconstructed as a separate 3DGS via OmniSplat4D Phase 1 (or loaded from an external SPZ file if the background was captured separately).
2. The foreground subject is represented as a 4DGS via OmniSplat4D Phase 2, trained using CorridorKey-quality alpha masks from the source footage.
3. At render time, `concat_gaussians()` merges the background 3DGS and the deformed subject 4DGS into a single primitive set.
4. `radix_sort_by_depth()` sorts the combined set by depth relative to the desired viewpoint.
5. `render_frame()` produces the final composite via unified alpha-blended rasterization.

This pipeline renders a geometrically-correct composite from any desired camera position — not just the original capture viewpoint. The output is a free-viewpoint composite.

### **6.3. Open Research Questions**

- **Scale calibration between source and destination environments.** If the foreground subject was captured in one physical location and the background environment is a separate 3DGS reconstruction from a different location, the metric scales of the two Gaussian representations must be aligned before concatenation is geometrically valid. The shared COLMAP world frame that makes OmniSplat4D's internal concatenation valid does not automatically extend to externally-sourced background 3DGS files. A scale registration step is required. The robustness and accuracy of this registration for practical VFX scenarios is unknown.

- **Lighting coherence.** The foreground 4DGS is trained on frames lit by the source environment. The destination background 3DGS reflects a different lighting environment. The DC and AC color coefficients of both primitive sets encode their respective lighting conditions. Naive concatenation will produce a composite where the foreground subject is visibly lit differently from the background environment. Correcting this requires either a relighting pass (see Section 7) or constraining the training of both representations to a shared lighting model. Neither approach is currently implemented.

- **Rendering resolution and real-time performance.** The unified radix sort over a merged primitive set of potentially millions of Gaussians (background 3DGS with several million primitives plus subject 4DGS with several hundred thousand) has not been benchmarked on the RTX 3060 12GB hardware target for this use case. The current Phase 3 design is optimized for the OmniSplat4D-internal compositing case. Performance characterization for the extended VFX compositing use case is unknown.

## **7. Integration Component 4 — Physically-Based Relighting from 3DGS Illumination**

### **7.1. Concept**

The most speculative of the four integration components, and the most research-intensive. Green screen spill correction — the process of removing the greenish cast from the subject's hair, skin, and clothing caused by light reflecting off the green screen — is currently performed as a 2D heuristic color grade in compositing software. The physically-correct approach requires computing the incident light from the actual destination background environment and applying it to the foreground subject's surface.

OmniSplat4D's static 3DGS background reconstruction encodes the radiance of the environment at every Gaussian primitive. In principle, this radiance field can be queried to compute the incident illumination at any 3D point in the scene. If the foreground subject's 3D geometry (from the 4DGS) and approximate surface properties are known, the incident illumination from the 3DGS background can be computed per-surface-point and applied as a relighting pass on top of CorridorKey's foreground color output.

This would replace the 2D spill correction heuristic with a physically-grounded relighting computation derived from the actual geometry of both scene elements.

### **7.2. Open Research Questions**

This component is substantially more research-intensive than Components 1–3 and raises more questions than it currently answers:

- **Radiance extraction from 3DGS.** 3DGS representations store appearance (via spherical harmonic or DC+AC color coefficients) but not explicit illumination. Decomposing the stored appearance into incident illumination, surface albedo, and surface normal is an inverse rendering problem that is not solved by the standard 3DGS training objective. Novel approaches such as GaussianShader or Relightable 3DGS are actively researching this decomposition, but no consensus method exists.

- **Dynamic subject surface properties.** Relighting requires knowledge of the subject's BRDF (bidirectional reflectance distribution function). The 4DGS model does not learn surface normals or material properties — it learns appearance directly. Obtaining surface normals from Gaussian primitives requires either a post-processing normal estimation step or a modified training objective. This is an active research area.

- **Computational cost.** Physically-based relighting computations, even approximations, are significantly more expensive than the direct rasterization pass. Whether this component is feasible at interactive rates on the RTX 3060 is unknown.

This component should be considered aspirational. It is included to indicate the upper bound of what geometrically-grounded compositing could provide, not to imply near-term implementability.

## **8. Structural Implications for the Current OmniSplat4D Codebase**

The following observations apply to Phases 1–5 of the current sprint roadmap. None require scope changes to those sprints, but they represent design decisions that should not be inadvertently closed off.

### **8.1. What the Current Architecture Already Gets Right**

The `GaussianCheckpoint` dataclass ([src/omnisplat4d/core/types.py](../../src/omnisplat4d/core/types.py)) contains an `opacities` field of shape `(N, 1)`. This is the data required to derive per-pixel alpha from the Gaussian field. No structural change is needed.

`concat_gaussians()` ([src/omnisplat4d/composite/renderer.py](../../src/omnisplat4d/composite/renderer.py)) is fully implemented and correctly merges two `GaussianCheckpoint` instances before rasterization. The Gaussian-level compositing primitive already exists.

The shared COLMAP world coordinate frame — established by `sfm/initializer.py` and respected by both `static_trainer.py` and `dynamic_trainer.py` — is the foundational geometric contract that makes multi-source Gaussian concatenation valid. This must be preserved.

### **8.2. The One Structural Gap**

`FrameBatch` ([src/omnisplat4d/core/types.py](../../src/omnisplat4d/core/types.py)) currently carries only `camera_dirs` (RGB frame directories) and `mask_dirs` (binary mask directories). It has no slot for continuous linear alpha. If CorridorKey's alpha output is to flow through the training pipeline as a training signal for the photometric loss, an `alpha_dirs` field would be required:

```python
alpha_dirs: Optional[list[Path]] = None  # Linear float alpha, per camera
```

This is a one-line addition to the dataclass. It should be added only when the Phase 2 training loop is implemented and the question of how to incorporate a continuous alpha into the gsplat photometric loss has been resolved. The loss computation when alpha is non-binary is a research question — the standard L1 + SSIM loss operates on RGB pixels, and incorporating a continuous foreground alpha requires either alpha-compositing the prediction before loss computation or adding a dedicated alpha consistency term.

### **8.3. Functions Not in the Current Sprint Plan**

Two thin wrapper functions become available naturally after Sprint 5 without requiring a dedicated sprint:

`render_background_at_pose(static_checkpoint, intrinsics, extrinsics) → np.ndarray` — calls `render_frame()` with only the static `GaussianCheckpoint`. Used to generate synthetic background renders for the hint generation pipeline described in Section 4.

`extract_alpha_matte(dynamic_checkpoint, intrinsics, extrinsics) → np.ndarray` — calls gsplat's rasterizer in alpha-output mode rather than RGB-output mode. The accumulated opacity channel is a standard output of the front-to-back transmittance accumulation already implemented in the compositing pass; this function extracts it as a standalone float image.

Both functions are approximately twenty lines each. They have no design complexity. They become implementable the moment `render_frame()` is complete.

## **10. Conclusion**

The architectural foundation of OmniSplat4D is geometrically compatible with the long-term vision described in this document. The shared world coordinate frame, the `GaussianCheckpoint` opacity field, the implemented `concat_gaussians()` primitive, and the design of `render_frame()` as a general-purpose rasterization call all point in the right direction without requiring retrofit.

The near-term roadmap — Sprints 3 through 5, implementing `train_static()`, `train_dynamic()`, `radix_sort_by_depth()`, and `render_frame()` — is not a detour from this vision. Completing those sprints is a prerequisite for it. The CorridorKey integration cannot be built on stubs.

The primary uncertainties are empirical, not architectural. They require footage, measurements, and iteration. They cannot be resolved by design documents.

The secondary uncertainty is the evolving state of CorridorKey itself. The repository at https://github.com/nikopueringer/CorridorKey.git is not a frozen dependency. Any implementation effort derived from this document should treat CorridorKey as a moving target and evaluate its current state — including active forks and community extensions — before committing to an integration approach.
